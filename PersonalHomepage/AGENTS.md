# AGENTS.md

## 项目目标

本项目用于实现一个“学术/个人主页风格”的个人网站。

当前阶段目标：

1. 高保真复刻参考站的整体信息架构、视觉层级、版式比例、交互节奏与页面组织方式。
2. 暂时使用“占位内容 / 临时文案 / 示例数据”填充页面，使页面结构、长度、密度、节奏接近参考站。
3. 后续逐步将占位内容替换为我自己的真实信息。

重要约束：

- 可以参考目标站点的布局、结构、组件拆分、视觉节奏与交互方式。
- 不要直接复制参考站的受版权保护内容，包括但不限于：原文案、图片、头像、简历 PDF、论文列表原文、下载文件、图标素材、站点源码、站点 CSS/JS。
- 所有临时内容都应使用“占位文本、假数据、我提供的数据、或模型生成的中性示例文案”。
- 所有资源文件必须是项目内自有资源、开源可商用资源，或明确标注为占位素材。

## 工作方式

1. 先阅读本文件，再阅读与当前任务最相关的文件。
2. 先给出一个简短计划，再开始改动。

4. 改动后运行必要检查，再汇总结果。
5. 回复时说明：

不要：

- 不要随意新增生产依赖，除非确有必要并说明原因。
- 不要未经说明就修改项目整体技术栈。
- 不要删除我已有内容，除非该内容已被新结构明确替代。

## 当前产品定位

这是一个“可逐步替换内容”的个人主页模板项目，不是一次性静态仿站。

实现上必须满足：

- 样式层与内容层分离
- 内容尽量数据化
- 组件可复用
- 后续替换个人信息时尽量只改数据文件，不改布局代码

---

## 页面与信息架构

默认按以下页面结构组织，除非我明确要求调整：

### 首页

首页建议包含：

- Hero / 首屏简介
- About / 个人简介
- Research Interests / 研究方向或兴趣方向
- Education / 教育经历
- Publications or Projects / 论文、项目、作品
- News / Updates / 动态（可选）
- Awards / Services / Experience（按需要）
- Contact / 联系方式
- Footer

### 结构要求

- 页面应尽量保持清晰的垂直阅读流。
- 每个 section 都应是独立组件。
- section 顺序允许调整，但必须保持整体风格统一。
- 页面应支持未来继续扩展为：
  - publications 独立页
  - projects 独立页
  - CV 下载
  - blog / notes
  - 中文 / 英文双语

## 目录约定

默认目录结构如下；若仓库已有结构，尽量贴合现有目录，不要机械重排：

- `src/app/`：页面与路由
- `src/components/`：可复用组件
- `src/components/sections/`：首页各 section
- `src/components/ui/`：基础 UI 组件
- `src/content/`：页面内容数据
- `src/content/site.ts`：站点主数据
- `src/content/publications.ts`：论文或项目列表
- `src/content/profile.ts`：个人资料
- `public/`：静态资源
- `public/images/`：自有图片或占位图片
- `public/files/`：自有下载文件

规则：

- 布局代码不要夹杂大段硬编码文案。
- 文案、链接、列表数据应尽量放在 `src/content/` 下统一管理。
- 不要把参考站 URL、站点抓取内容、原站资源地址写死在生产代码里。

---

## 视觉与交互复刻原则

目标是“复刻风格与结构”，不是“复制源码”。

应优先复刻：

- 信息层级
- 页面节奏
- 模块顺序
- 留白比例
- 字号层级
- 卡片/列表/时间线等展示方式
- 导航结构
- 响应式行为
- hover / transition / anchor scroll 等交互感受

不要直接照搬：

- 原站 HTML/CSS/JS 源码
- 原站图片、PDF、favicon、logo
- 原站专有文案
- 原站字体文件（除非确认授权可用）

实现要求：

- 使用原创组件代码重建页面。
- 保持代码可读性，不要输出“为了像而像”的混乱样式。
- 优先抽象为可复用 section，而不是堆砌一页长 JSX。
- 响应式必须完善，至少覆盖 desktop / tablet / mobile。

## 内容迁移策略

内容替换分为两个阶段：

### 阶段 1：占位复刻

- 使用占位标题、占位简介、占位经历、占位论文/项目数据。
- 占位内容的长度、段落数、列表密度可参考目标站视觉效果。
- 所有占位内容必须集中存放在 `src/content/`，便于后续替换。

### 阶段 2：逐步替换为我的信息

- 我提供真实资料后，优先只更新数据文件。
- 尽量避免为替换内容而大改布局。
- 替换时保持页面整体风格稳定。
- 当真实内容长度与占位差异较大时，先最小调整布局，再更新样式。

## 样式规范

- 优先使用现有设计 token；若无，则保持统一的 spacing、radius、shadow、container 宽度。
- 不要随意混用多套视觉语言。
- 不要引入过多颜色；整体保持简洁、专业、克制。
- 动效应轻量，不花哨，不影响阅读。
- 字体、字号、间距应服务于信息展示，而不是炫技。

---

## 组件设计规范

每个 section 组件应满足：

- 单一职责
- 易于替换数据
- props 清晰
- 不直接耦合到参考站内容
- 样式尽量内聚
- 命名直观

能抽离就抽离：

- SectionHeader
- SocialLinks
- PublicationList / ProjectList
- Timeline
- ProfileCard
- ContactCard
- Footer

---

## 数据规范

内容数据建议使用结构化对象，不要在 JSX 里散落硬编码文本。

示例原则：

- 个人资料：姓名、标题、简介、邮箱、社交链接
- 教育经历：学校、学位、时间、描述
- 项目/论文：标题、摘要、链接、标签、时间
- 联系方式：邮箱、GitHub、X、Google Scholar、LinkedIn 等

新增数据字段时：

- 先复用已有结构
- 只有确实需要时再扩展 schema
- 保持字段命名一致

## 当我说“继续复刻”时

默认理解为：

1. 在不复制受保护原内容的前提下，继续逼近参考站的布局和交互。
2. 优先处理最影响相似度的部分：
   - 页面结构
   - section 顺序
   - 字体层级
   - 间距比例
   - 卡片/列表表现
   - 导航与锚点
3. 使用占位内容维持视觉完整性。

---

## 当我说“开始替换成我的信息”时

默认理解为：

1. 优先修改 `src/content/` 下的数据文件。
2. 仅在必要时微调组件。
3. 保持页面整体视觉尽量不变。
4. 替换后检查：
   - 文本是否溢出
   - 链接是否有效
   - section 是否仍然平衡
   - 中英文混排是否自然
